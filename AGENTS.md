## 核心原则

**没有文档，不写代码；没有确认，不进入下一阶段**

---

## Phase-Gate 工作流程

所有"开始新功能开发"的请求，默认执行以下五阶段流程：

### Phase 0: 需求澄清

- **禁止**: 编写任何技术方案或代码
- **必须**: 通过提问澄清业务背景、用户价值、验收标准
- **输出**: 需求澄清备忘录（非正式，用于对齐认知）
- **闸门**: 用户回复 **"确认进入 Phase 1"**

### Phase 1: 需求规格(SRS)

- **禁止**: 编写技术设计或代码
- **必须**: 输出符合 IEEE 830 标准的软件需求规格说明书
- **格式**: Markdown，包含功能/非功能需求、验收标准、优先级
- **闸门**: 用户回复 **"确认通过 SRS，进入 Phase 2"**

### Phase 2: 技术设计(TDD)

- **禁止**: 编写业务逻辑代码
- **必须**: 输出技术设计文档（架构图、接口定义、数据库设计、测试策略）
- **格式**: 包含 Mermaid 图表、OpenAPI 规格、关键算法伪代码
- **闸门**: 用户回复 **"确认通过设计，进入 Phase 3"**

### Phase 3: 开发实现

- **禁止**: 擅自修改已确认的需求或设计
- **必须**: 按设计文档编写代码，附带单元测试
- **输出**: 可运行的代码 + 测试用例 + 代码注释
- **闸门**: 用户回复 **"确认代码通过，进入 Phase 4"**

### Phase 4: 验收交付

- **必须**: 提供部署指南、使用说明、变更日志
- **输出**: 完整的交付物清单
- **结束**: 用户回复 **"确认交付完成"**

---

## 闸门规则

1. 每阶段必须收到"确认进入 Phase X"或"确认通过"才推进
2. 每轮回复必须显示 `[Phase X: 阶段名]` 标识
3. **禁止混合阶段内容**

---

## 触发词

| 触发词 | 动作 |
|--------|------|
| "开始新功能开发" / "开发 xxx功能" | 进入 **Phase 0** |
| "继续" / "下一阶段" | 推进到下一 Phase |
| "修改需求" | 进入变更控制流程 |
