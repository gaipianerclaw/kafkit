## 1. 引言

### 1.1 目的

本文档详细描述 Kafkit 桌面应用程序的功能需求、非功能需求和验收标准，作为设计、开发、测试和验收的依据。

### 1.2 范围

Kafkit 是一款面向开发人员和运维人员的跨平台桌面工具，用于连接和管理 Apache Kafka 集群，执行消息生产和消费操作。

### 1.3 定义和缩略语

| 术语 | 定义 |
|------|------|
| Kafka | Apache Kafka 分布式流处理平台 |
| Topic | Kafka 消息主题 |
| Partition | 分区，Topic 的水平切分单元 |
| Consumer Group | 消费者组，一组协同消费 Topic 的消费者 |
| Lag | 消费延迟，未消费消息数 |
| SASL | 简单认证和安全层 |
| SSL/TLS | 安全传输层协议 |
| MVP | 最小可行产品 (Minimum Viable Product) |
| Tauri | 构建跨平台桌面应用的框架 |

### 1.4 参考资料

- Apache Kafka 官方文档: https://kafka.apache.org/documentation/
- Tauri 框架文档: https://tauri.app/
- librdkafka 文档: https://github.com/confluentinc/librdkafka

## 2. 总体描述

### 2.1 产品视角

Kafkit 是一个独立的桌面应用程序，通过原生 Kafka 客户端库与 Kafka 集群交互。采用 Tauri 框架构建，前端使用 Web 技术，后端使用 Rust 实现。

**系统架构图:**

```
┌─────────────────────────────────────────┐
│           Kafkit 桌面应用                │
│  ┌──────────────┐    ┌──────────────┐  │
│  │  Web 前端 UI │◄──►│  Rust 后端   │  │
│  │  (React/Vue) │ IPC │  (rdkafka)   │  │
│  └──────────────┘    └──────┬───────┘  │
│                             │          │
│                      ┌──────▼───────┐  │
│                      │  加密本地存储 │  │
│                      └──────────────┘  │
└─────────────────────────────────────────┘
                   │
                   │ librdkafka
                   ▼
          ┌─────────────────┐
          │ Apache Kafka    │
          │ 集群            │
          └─────────────────┘
```

### 2.2 用户特征

| 用户类型 | 技术背景 | 使用场景 |
|----------|----------|----------|
| 后端开发 | 熟悉 Kafka 和命令行 | 开发调试、消息验证 |
| 运维工程师 | 熟悉集群管理 | 故障排查、配置检查 |
| 测试工程师 | 基础 Kafka 知识 | 测试数据构造、结果验证 |

### 2.3 运行环境

| 平台 | 最低版本 | 说明 |
|------|----------|------|
| macOS | 11.0+ | Intel & Apple Silicon |
| Windows | Windows 10 1809+ | x64 |
| Linux | Ubuntu 20.04+ | x64, GTK 3.0+ |

### 2.4 约束和限制

1. **安全性**：连接凭证必须加密存储
2. **性能**：单 Topic 消费时延迟 < 100ms
3. **兼容性**：支持 Kafka 2.0+ 版本
4. **扩展性**：数据格式支持可插拔设计

### 2.5 假设和依赖

- 用户拥有 Kafka 集群的有效访问凭证
- 网络可达 Kafka 集群（Broker 地址可解析、端口可连接）
- 对于 SASL/GSSAPI，系统已配置 Kerberos 环境

## 3. 功能需求

### 3.1 连接管理 (CM)

#### CM-001: 多集群配置管理
**优先级**: 高  
**描述**: 用户可添加、编辑、删除多个 Kafka 集群连接配置  
**输入**: 集群名称、Broker 地址列表、认证信息  
**输出**: 保存的配置项  
**验收标准**:
- 支持至少 20 个集群配置
- 配置增删改查操作响应 < 500ms
- 配置变更后立即持久化

#### CM-002: 连接信息加密存储
**优先级**: 高  
**描述**: 所有敏感信息（密码、证书）必须加密存储  
**验收标准**:
- 使用系统密钥链或 AES-256 加密本地文件
- 密码不以明文形式出现在内存转储中

#### CM-003: 认证方式支持
**优先级**: 高  
**描述**: 支持多种 Kafka 安全认证方式  
**支持的认证**:
| 认证方式 | 配置项 |
|----------|--------|
| PLAINTEXT | 无 |
| SASL/PLAIN | username, password |
| SASL/SCRAM-SHA-256/512 | username, password |
| SASL/GSSAPI (Kerberos) | principal, keytab, service name |
| SSL | ca cert, client cert, client key |

**验收标准**:
- 每种认证方式可成功建立连接
- 认证失败时显示清晰的错误信息

#### CM-004: 连接健康检查
**优先级**: 中  
**描述**: 检测与 Kafka 集群的连接状态  
**验收标准**:
- 连接/断开操作有明确反馈
- 连接异常时显示状态指示器

---

### 3.2 Topic 管理 (TM)

#### TM-001: Topic 列表展示
**优先级**: 高  
**描述**: 列出集群中的所有 Topic  
**展示字段**: Topic 名称、分区数、副本因子  
**验收标准**:
- 支持 1000+ Topic 的流畅展示
- 支持按名称搜索/过滤
- 列表可刷新

#### TM-002: Topic 基本信息查看
**优先级**: 高  
**描述**: 查看选定 Topic 的详细信息  
**展示信息**:
- 分区详情（分区号、Leader、Replicas、ISR）
- Topic 级别配置（retention.ms, cleanup.policy 等）
- 各分区的消息数量估算

#### TM-003: 创建 Topic
**优先级**: 高  
**描述**: 创建新的 Kafka Topic  
**输入**: Topic 名称、分区数、副本因子、可选配置  
**约束**:
- Topic 名称符合 Kafka 命名规范
- 分区数 ≥ 1, 副本因子 ≥ 1 且 ≤ Broker 数量
- 创建失败时显示错误原因

#### TM-004: 删除 Topic
**优先级**: 高  
**描述**: 删除指定的 Topic  
**验收标准**:
- 删除前必须二次确认
- 显示删除操作结果
- 处理删除权限不足的情况

---

### 3.3 消息消费 (MC)

#### MC-001: 实时消息消费
**优先级**: 高  
**描述**: 实时消费选定 Topic 的消息  
**功能**:
- 从最新位置开始消费
- 显示消息流入速率
- 可暂停/恢复消费
- 可清空当前视图

**验收标准**:
- 消息展示延迟 < 500ms
- 支持至少 100 msg/s 的显示

#### MC-002: 历史消息查看
**优先级**: 高  
**描述**: 查看 Topic 中的历史消息  
**功能**:
- 按分区选择
- 指定起始偏移量或时间戳
- 分页/滚动加载
- 每页默认 100 条，可调整

#### MC-003: 消息格式化显示
**优先级**: 高  
**描述**: 以不同格式解析和显示消息内容  
**支持格式**:
| 格式 | 说明 |
|------|------|
| Plain Text | 原始文本 |
| JSON | 语法高亮、格式化 |
| CSV | 表格形式展示 |

**验收标准**:
- 自动检测 JSON 格式
- 大消息（>1MB）可折叠显示
- 消息内容可复制

#### MC-004: 消息元信息展示
**优先级**: 中  
**描述**: 展示消息的完整元信息  
**展示字段**: offset, partition, timestamp, key, headers, size

---

### 3.4 消息生产 (MP)

#### MP-001: 单条消息发送
**优先级**: 高  
**描述**: 向指定 Topic 发送单条消息  
**输入**:
- Topic 名称
- Partition（可选，默认自动分配）
- Key（可选）
- Value（消息内容）
- Headers（可选键值对）

**验收标准**:
- 发送成功显示确认
- 发送失败显示错误详情
- 支持消息重发

#### MP-002: 批量消息发送
**优先级**: 高  
**描述**: 批量发送多条消息  
**功能**:
- 文本框输入多条消息（每行一条或 JSON 数组）
- 配置发送速率（条/秒）
- 显示发送进度和统计

**验收标准**:
- 支持 1000+ 条消息的批量发送
- 可配置发送间隔防止压垮集群
- 部分失败时显示成功/失败统计

#### MP-003: 消息格式支持
**优先级**: 高  
**描述**: 支持不同格式的消息内容  
**支持**: Plain Text / JSON / CSV  
**验收标准**:
- JSON 格式输入时进行语法验证
- 提供格式化辅助（JSON 美化）

---

### 3.5 Consumer Group 管理 (CG)

#### CG-001: Consumer Group 列表
**优先级**: 高  
**描述**: 展示集群中的所有 Consumer Group  
**展示字段**: Group ID、状态、Member 数量、Coordinator  
**验收标准**:
- 支持 100+ Group 的展示
- 支持按名称搜索

#### CG-002: Group 详情查看
**优先级**: 高  
**描述**: 查看 Consumer Group 的详细信息  
**展示信息**:
- 订阅的 Topics
- 各 Partition 的消费进度（Current Offset, Log End Offset, Lag）
- Group Members 详情

#### CG-003: Lag 监控
**优先级**: 高  
**描述**: 显示消费延迟信息  
**功能**:
- 按 Topic 汇总 Lag
- 按 Partition 展示 Lag
- 高 Lag 预警（可配置阈值）

#### CG-004: 偏移量重置
**优先级**: 中  
**描述**: 重置 Consumer Group 的消费偏移量  
**支持策略**:
- 重置到最早 (earliest)
- 重置到最新 (latest)
- 重置到指定时间点
- 重置到指定偏移量

**验收标准**:
- 重置前二次确认
- 显示重置结果
- 处理 Group 处于活跃状态的情况

---

### 3.6 扩展性需求 (EXT)

#### EXT-001: 数据格式插件机制
**优先级**: 中  
**描述**: 预留消息编解码扩展接口  
**说明**: 架构设计需支持后续添加 Avro、Protobuf 等格式，不强制在 MVP 实现

#### EXT-002: 消息搜索预留
**优先级**: 低  
**描述**: 架构预留消息过滤和搜索能力  
**说明**: 当前版本不实现，但消费模块设计需考虑后续扩展

## 5. 接口需求

### 5.1 用户界面

**主要界面**:
1. **欢迎/启动页**: 选择已有连接或创建新连接
2. **连接配置页**: 表单形式配置连接参数
3. **主工作区**: 左侧导航（Topics/Groups/设置），右侧内容区
4. **Topic 详情页**: 分区信息、消费/生产入口
5. **消费页面**: 消息列表（类日志查看器风格）
6. **生产页面**: 消息编辑和发送
7. **Group 管理页**: 列表和详情视图

**设计原则**:
- 遵循各平台原生设计语言
- 深色/浅色主题切换
- 响应式布局适配不同窗口尺寸

### 5.2 软件接口

**Kafka 客户端**: 
- Rust `rdkafka` crate (librdkafka 绑定)
- 支持完整的 Kafka 协议

**本地存储**:
- Tauri Store API / 加密文件
- 配置持久化格式：JSON（加密后）
